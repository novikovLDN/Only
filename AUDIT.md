# Аудит проекта: Telegram Habit Bot

**Дата:** 2026-03-25
**Версия:** 1.0.0

---

## 1. Суть проекта

**Telegram-бот для трекинга привычек** с премиум-подпиской, геймификацией и крипто-оплатой.

### Основные возможности

| Функция | Описание |
|---------|----------|
| **Привычки** | Создание/редактирование/удаление привычек с расписанием (день недели + время) |
| **Напоминания** | Каждые 60 секунд проверяет расписание, отправляет уведомления с учётом часового пояса |
| **Логирование** | Пользователь отмечает привычку как "Выполнено" или "Пропущено" |
| **Геймификация** | 10 уровней, XP за выполнение, 50+ достижений (серии, рекорды, рефералы) |
| **Premium** | Подписка через карту (YooKassa) и криптовалюту (2328.io) — 99-829₽/мес |
| **Реферальная программа** | Инвайт-ссылка `/start ref_<id>`, +7 дней Premium за приглашённого |
| **Админ-панель** | Поиск пользователей, выдача/отзыв Premium, скидки, удаление |
| **Мультиязычность** | Русский, Английский, Арабский |

### Архитектура

```
Handlers (aiogram роутеры) → Services (бизнес-логика) → Repositories (доступ к данным) → Models (SQLAlchemy ORM) → PostgreSQL (asyncpg)
```

**Стек:** Python 3.12, aiogram 3.13, FastAPI, SQLAlchemy 2.0 (async), APScheduler, PostgreSQL, Docker, Railway

## 2. Аудит тестов

### Статус: ТЕСТЫ ОТСУТСТВУЮТ

В проекте **нет тестов вообще** — ни unit, ни integration, ни e2e. Нет `tests/` директории, нет `pytest` в зависимостях, нет CI/CD пайплайна.

### Критичность: ВЫСОКАЯ

Без тестов невозможно:
- Гарантировать корректность платёжной логики (деньги пользователей)
- Безопасно рефакторить код
- Ловить регрессии при обновлении зависимостей
- Проверять граничные случаи (часовые пояса, скидки, XP)

### Рекомендации по тестированию

**Приоритет 1 — Платежи (критично):**
- `test_calculate_price_with_discount()` — проверка расчёта скидок, граничные случаи (0%, 99%, истёкшая скидка)
- `test_extend_premium()` — продление подписки, переполнение дат, уже-premium пользователь
- `test_process_crypto_webhook()` — валидная/невалидная подпись, дубликат оплаты, отмена
- `test_verify_webhook_signature()` — HMAC-валидация, timing-safe сравнение

**Приоритет 2 — Бизнес-логика:**
- `test_habit_creation()` — лимит для бесплатных (3 привычки), валидация названия (1-100 символов)
- `test_reminder_timezone()` — корректное сопоставление времени в разных часовых поясах
- `test_referral_flow()` — создание реферала, предотвращение самореферала, выдача бонуса
- `test_achievement_unlock()` — триггеры достижений (серии, объёмы)

**Приоритет 3 — Инфраструктура:**
- `test_db_migrations()` — идемпотентность миграций
- `test_session_management()` — корректный commit/rollback в middleware
- `test_scheduler_jobs()` — расписание задач, обработка ошибок

## 3. Аудит безопасности

### 3.1. КРИТИЧЕСКИЕ ПРОБЛЕМЫ

#### [CRITICAL-1] Захардкоженный ADMIN_ID
**Файл:** `app/config.py:47`
```python
ADMIN_ID = 6214188086
```
**Риск:** При утечке репозитория атакующий знает ID администратора. Если админский Telegram-аккаунт будет скомпрометирован — полный контроль над ботом.
**Рекомендация:** Перенести в переменные окружения:
```python
ADMIN_ID = int(os.environ.get("ADMIN_ID", "0"))
```

#### [CRITICAL-2] Захардкоженный crypto_project_id
**Файл:** `app/config.py:24`
```python
crypto_project_id: str = "8f7c14f1-2473-4652-9f7a-5c04693b40af"
```
**Риск:** Идентификатор проекта 2328.io в открытом коде. Может использоваться для таргетированных атак на платёжный шлюз.
**Рекомендация:** Перенести в `.env`.

#### [CRITICAL-3] SQL-инъекция в db_autofix.py
**Файл:** `app/db_autofix.py:18-28`
```python
sql = text(f"""
    ...WHERE table_name = '{table}' AND column_name = '{column_name}'
    ...EXECUTE 'ALTER TABLE {table} ADD COLUMN {column_def}';
""")
```
**Риск:** Параметры `table` и `column_def` вставляются в SQL через f-string без санитизации. Хотя вызывается только с хардкод-значениями, это опасный паттерн — при рефакторинге может стать вектором атаки.
**Рекомендация:** Использовать `IF NOT EXISTS` синтаксис PostgreSQL напрямую (как уже сделано в `database.py`), или валидировать имена через whitelist.

### 3.2. ВЫСОКИЙ РИСК

#### [HIGH-1] Отсутствие Rate Limiting
Нет ограничений на:
- Создание привычек (DoS на БД)
- Запросы на создание платежей (флуд платёжного шлюза)
- Команды бота (спам)
**Рекомендация:** Добавить `aiogram-throttling` middleware или кастомный rate limiter (например, 5 запросов/сек на пользователя).

#### [HIGH-2] MemoryStorage для FSM
**Файл:** `app/main.py:41`
```python
dp = Dispatcher(storage=MemoryStorage())
```
**Риск:** При рестарте бота все FSM-состояния теряются. Пользователь, начавший создание привычки, получит ошибку.
**Рекомендация:** Перейти на `RedisStorage` для персистентности. Минимально — обрабатывать потерю состояния gracefully.

#### [HIGH-3] Жёсткий курс RUB → USD
**Файл:** `app/services/crypto_service.py:158-160`
```python
def _rub_to_usd(price_rub: float) -> float:
    return round(price_rub / 100, 2)
```
**Риск:** Фиксированный курс 100₽/$. При изменении реального курса пользователи переплачивают или бот теряет деньги.
**Рекомендация:** Использовать API курса валют (ЦБ РФ, exchangerate-api) или задавать курс через переменную окружения.

#### [HIGH-4] Нет логирования действий администратора
Админ может удалять пользователей, выдавать премиум и скидки без audit trail.
**Рекомендация:** Логировать все админ-действия в отдельную таблицу `admin_audit_log`.

### 3.3. СРЕДНИЙ РИСК

#### [MEDIUM-1] Предсказуемые реферальные ссылки
Формат: `/start ref_<telegram_id>`. Telegram ID — последовательные числа, легко перебираются.
**Рекомендация:** Использовать UUID или HMAC-подпись для реферальных кодов.

#### [MEDIUM-2] Нет валидации webhook URL
**Файл:** `app/health.py` — FastAPI endpoint без CORS, без IP-фильтрации.
**Риск:** Кто угодно может отправить запрос на `/webhook/crypto`.
**Смягчение:** HMAC-подпись проверяется (хорошо), но стоит добавить IP-whitelist от 2328.io.

#### [MEDIUM-3] PII хранится без шифрования
Telegram ID, имена пользователей, история привычек, платёжные данные — всё в plaintext в PostgreSQL.
**Рекомендация:** Как минимум, шифровать `first_name` и `username` на уровне приложения.

### 3.4. ЧТО СДЕЛАНО ХОРОШО

- HMAC-SHA256 для подписи крипто-запросов с `hmac.compare_digest()` (защита от timing attack)
- Параметризованные SQL-запросы через SQLAlchemy ORM (нет SQL-инъекций в основном коде)
- API-ключи в переменных окружения (кроме ADMIN_ID и project_id)
- Whitelist для языков и часовых поясов
- Идемпотентная обработка webhook (проверка `payment.status in ("paid", "completed")`)
- CASCADE DELETE на FK — нет orphan-записей
- Маскирование DATABASE_URL в логах

## 5. Резюме

### Сильные стороны
- Чистая слоистая архитектура (handlers → services → repos → models)
- Полностью асинхронный стек
- Грамотная криптографическая подпись платежей
- Продуманная геймификация (50+ достижений, уровни, XP)
- Мультиязычность

### Критичные проблемы
1. **Нет тестов** — главный риск проекта
2. **Захардкоженные секреты** (ADMIN_ID, crypto_project_id)
3. **Нет rate limiting** — уязвимость к DoS
4. **Нет миграций** (Alembic) — риск потери данных при изменении схемы
5. **Фиксированный курс валют** — финансовый риск

### Оценка зрелости: 5/10
Проект имеет хорошую архитектурную основу, но ему критически не хватает тестов, мониторинга и CI/CD для production-готовности.
